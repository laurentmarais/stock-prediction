Stock Prediction Project Progress

Date: March 25, 2026

Current Objective

Build a chart-first stock scenario application that:

- uses free data sources by default,
- keeps data providers modular so paid sources can be added later,
- supports replay from a previous point in time,
- and shows forecast distributions and realized outcomes in an easy-to-review UI.

Provider modularity design

- Each provider category is isolated behind a narrow interface.
- The rest of the app does not talk directly to a vendor implementation.
- Future paid provider upgrades should mostly require:
	- a new adapter file,
	- a registry entry,
	- optional env-var configuration.

Important Fixes Made

1. Python 3.13 compatibility

- Removed `pandas-datareader` dependency from the macro provider path because it relied on `distutils`, which breaks on Python 3.13.
- Replaced it with a direct FRED CSV fetch in the FRED provider adapter.

2. Yahoo Finance MultiIndex normalization

- Yahoo market data sometimes returned a `MultiIndex` column shape even for a single ticker.
- Added normalization in the Yahoo market provider so downstream code always sees a stable schema.

3. Replay forecast endpoint bug

- Fixed the 1-dimensional sampling bug that came from provider output shape mismatches.

4. Regime history timestamp bug

- Fixed a bug where integer positions were incorrectly used as timestamps in the returned regime history.
- Regime history now carries explicit timestamps and separate integer positions for sampling.

Current Project State

What is working now

- modular backend and frontend scaffold
- free-source data flow
- replay-aware chart and forecast endpoints
- live frontend build
- replay comparison UI
- regime-aware forecast metadata
- regime history strip in the UI

What is still basic

- forecast model is still a lightweight heuristic engine, not a full ML or HMM-based probabilistic model
- no database persistence yet beyond the code scaffold assumptions
- no calibration dashboard yet
- no saved replay sessions or watchlists yet
- no true point-in-time fundamentals dataset, only point-in-time-like behavior via historical chart anchoring and current free-source snapshots

Recommended Next Steps

Priority 1: Improve model quality

- Replace the rule-based regime classifier with a true HMM-based regime model while preserving the current response contract.
- Add a simple feature-based drift adjustment so forecasts do not depend only on matched historical blocks.
- Add separate volatility modeling rather than relying only on sampled return blocks.

Priority 2: Add calibration and review tools

- Build a calibration view showing:
	- how often realized outcomes stay inside the 10-90 band,
	- how often they stay inside the 25-75 band,
	- realized error versus median path,
	- replay-by-replay evaluation.

Priority 3: Improve replay workflow further

- Add an on-chart replay anchor marker directly inside the chart area.
- Add realized path labeling on-chart.
- Add replay session controls such as stepping one bar forward or backward.

Priority 4: Prepare for stronger data sources

- Add placeholder provider modules for likely paid upgrades even if they are not yet implemented.
- Keep the provider contract strict enough that upgrades do not force broad refactors.

Priority 5: Operationalize the app

- Add persistent storage for forecasts, replay sessions, and cached chart responses.
- Add API tests.
- Add frontend fetch error handling and loading states.
- Add Docker runtime validation if container execution becomes the preferred path.

Recommended Immediate Next Task

The highest-value next engineering step is:

- implement an HMM-based regime classifier behind the existing regime interface,

because that improves forecast realism without forcing major API or UI redesign.

Short Handoff Summary

The project has moved from architecture notes into a working, replay-aware prototype with a live backend, a buildable frontend, modular free-source providers, regime-aware forecasting, and multiple replay comparison views. The next major milestone should focus on improving probabilistic forecast quality and calibration rather than adding more surface-level UI features.
